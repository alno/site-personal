Использование

Для использования SLF4J, classpath проекта должен включать библиотеку slf4j-api.jar, которая содержит классы фасада. Для использования логирования в коде рассмотрим следующий пример кода: import org.slf4j.Logger; import org.slf4j.LoggerFactory; class MyClass { private Logger log = LoggerFactory.getLogger( MyClass.class ); // 1. Объявляем переменную логгера ... log.debug( "..." ); // 2. Выводим строку в debug ... log.info( "Some object: {}", object ); // 3. Выводим в info строку с аргументом, подставляемым в нее ... log.error( "Error during some job!!", e ); // 4. Выводим в error ошибку, вместе с исключением } Теперь рассмотрим, что же в этом примере делалось:

  1. Объявление переменной логгера. Логгер имеет тип org.slf4j.Logger и создается вызовом метода getLogger() у класса org.slf4j.LoggerFactory. В качестве параметра методу передается класс, в этом случае имя логгера совпадает с полным именем класса. Также можно передать строку, которая будет именем логгера.
  2. Вывод строки в лог. Методом debug в лог выводится запись с уровнем DEBUG, в точности такая, как была передана в качестве аргумента. Аналогично, существуют методы info, warn и error.
  3. Вывод в лог строки с аргументом. В запись в логе вместо {} подставляется второй аргумент метода. Таким образом можно передать до двух параметров, большее число необходимо оборачивать в массив. Чем такая передача лучше, чем просто "Some object: " + object? Тем, что в случае, когда записи с уровнем info не выводятся в лог, преобразование объекта object в строку и склеивание с "Some object: " может не производится, что может дать выигрыш в производительности.
  4. Вывод в лог строки с исключением. Вторым аргументом метода может так же быть исключение. Причем, исключение можно добавлять не только к ошибкам, но и к любым записям.

Конфигурация

Теперь, о том, как подключить какую-то реализацию к SLF4J. Допустим, мы хотим использовать в качестве реализации простейшую, которая выводит все на стандартный вывод. Такая реализация входит в slf4j и называется simple. Чтобы ее использовать, достаточно всего-навсего добавить библиотеку slf4j-simple.jar в classpath приложения. Теперь, предположим, нам захотелось, отключить вывод логов вообще, причем сделать это максимально эффективно. В смысле, чтобы даже никаких проверок не выполнялось. Добавляем в classpath вместо slf4j-simple.jar библиотеку slf4j-nop.jar. И логов больше нет) Аналогично, если необходимо использовать log4j или JCL, необходимо добавиь в classpath библиотеки slf4j-log4j12.jar или slf4j-log4j13.jar, или slf4j-jcl.jar. Разумеется, потом необходимо сконфигурировать саму реализацию (если она конфигурируется, конечно). Вот и вся конфигурация.
